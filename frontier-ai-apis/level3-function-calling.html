Function Calling

3.1 The Big Idea ‚Äî Models That Call Functions

Instead of just generating text, the model can decide to call a function you define. It outputs structured JSON arguments, you execute the function, and feed the result back.

sequenceDiagram participant U as User participant A as Your App participant M as AI Model participant T as Your Function U->>A: "What's the weather in NYC?" A->>M: Send message + tool definitions M-->>A: tool_call: get_weather(location: "NYC") A->>T: Execute get_weather("NYC") T-->>A: {"temp": "72F", "condition": "Sunny"} A->>M: Send tool result back M-->>A: "The weather in NYC is 72F and Sunny!" A-->>U: Display answer

3.2 Defining Tools ‚Äî Side by Side

OpenAI Chat Completions 
{
  "model": "gpt-4o",
  "messages": [
    {"role": "user",
     "content": "Weather in NYC?"}
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "Get current weather",
        "parameters": {
          "type": "object",
          "properties": {
            "location": {
              "type": "string",
              "description": "City name"
            }
          },
          "required": ["location"]
        }
      }
    }
  ]
}
Wrapped in type: "function" + function: {...}
Anthropic Messages API 
{
  "model": "claude-sonnet-4-5",
  "max_tokens": 1024,
  "messages": [
    {"role": "user",
     "content": "Weather in NYC?"}
  ],
  "tools": [
    {
      "name": "get_weather",
      "description": "Get current weather",
      "input_schema": {
        "type": "object",
        "properties": {
          "location": {
            "type": "string",
            "description": "City name"
          }
        },
        "required": ["location"]
      }
    }
  ]
}
Flat structure. Uses input_schema not parameters.

Tool Definition Differences

Structure OAI Nested: tools[].function.parameters Ant Flat: tools[].input_schema
Type wrapper OAI Requires "type": "function" Ant No type wrapper needed
Schema field OAI parameters Ant input_schema

3.3 When the Model Calls a Tool ‚Äî Response Format

OpenAI Response with tool_call 
// Model response
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "get_weather",
          "arguments": "{\"location\":\"NYC\"}"
        }
      }]
    },
    "finish_reason": "tool_calls"
  }]
}

// Arguments are JSON string!
// Must JSON.parse() them
Anthropic Response with tool_use block 
// Model response
{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Let me check..."
    },
    {
      "type": "tool_use",
      "id": "toolu_abc123",
      "name": "get_weather",
      "input": {
        "location": "NYC"
      }
    }
  ],
  "stop_reason": "tool_use"
}

// Input is already parsed object!
// Can include text + tool_use

Critical Differences

Arguments OAI JSON string (needs parsing) Ant Parsed object (ready to use)
Mixed content OAI content is null when tool calling Ant Can have text + tool_use in same response
Stop reason OAI "finish_reason": "tool_calls" Ant "stop_reason": "tool_use"

3.4 Sending Tool Results Back

OpenAI 
// Append to messages:
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "{\"temp\":\"72F\"}"
}

// Special "tool" role
// Must match tool_call_id
Anthropic 
// Append to messages:
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_abc123",
      "content": "Temp: 72F, Sunny"
    }
  ]
}

// Uses "user" role with tool_result block
// Must match tool_use_id
Key difference: OpenAI has a dedicated "role": "tool", while Anthropic uses "role": "user" with a tool_result content block. Both require matching the call ID.

2.2 Building Multi-Turn ‚Äî Code

OpenAI Chat Completions 
{
  "model": "gpt-4o",
  "messages": [
    {"role": "system",
     "content": "You are a DNS expert."},
    {"role": "user",
     "content": "What is DNS?"},
    {"role": "assistant",
     "content": "DNS stands for..."},
    {"role": "user",
     "content": "How does resolution work?"}
  ]
}
// You manage the full history
// Each call = full conversation resent
Anthropic Messages API 
{
  "model": "claude-sonnet-4-5",
  "max_tokens": 1024,
  "system": "You are a DNS expert.",
  "messages": [
    {"role": "user",
     "content": "What is DNS?"},
    {"role": "assistant",
     "content": "DNS stands for..."},
    {"role": "user",
     "content": "How does resolution work?"}
  ]
}
// Same pattern - you manage history
// system prompt stays top-level
Both APIs are stateless ‚Äî you must send the entire conversation history with each request. This means your app must store and append messages. Token costs grow with each turn.

2.3 Game Changer: OpenAI Responses API (Stateful)

The Responses API eliminates the need to resend history. The server remembers the conversation. You just reference the previous response ID.

Turn 1 ‚Äî Initial Request

POST /v1/responses

{
  "model": "gpt-4o",
  "input": "What is DNS?",
  "instructions": "You are a DNS expert.",
  "store": true
}

// Response includes:
// "id": "resp_abc123"

Turn 2 ‚Äî Just Reference Previous

POST /v1/responses

{
  "model": "gpt-4o",
  "input": "How does resolution work?",
  "previous_response_id": "resp_abc123"
}

// No need to resend Turn 1!
// Server remembers context
// Saves tokens and bandwidth

Stateless vs Stateful Comparison

History management Stateless (Chat Completions/Messages): Client sends full history Stateful (Responses API): Server stores, client sends ID
Token cost per turn Grows linearly (re-sending everything) Constant (only new input)
Caching benefit Client-side only 40-80% server-side cache improvement
Anthropic Note: Anthropic Messages API remains stateless. However, it supports prompt caching ‚Äî you can mark frequently reused prompt prefixes to be cached server-side for up to 1 hour, reducing costs on repeated prefixes.
